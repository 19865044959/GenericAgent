# GA 记忆系统：原方案 vs 改进方案

**日期**: 2026-06-08

---

## 1. 原方案：纯手动 RAG

### 1.1 架构

```
┌─ 系统提示词 ─────────────────────────────────────┐
│  L1 (global_mem_insight.txt)                     │
│  + CONSTITUTION (insight_fixed_structure.txt)     │
│                                                   │
│  只列出 L2/L3/L4 的"钩子"：                        │
│  "L2: global_mem.txt (User Profile)"             │
│  "L3: skill_search / memory_cleanup_sop / ..."    │
│  "L4: L4_raw_sessions/ historical sessions"      │
└───────────────────────────────────────────────────┘
                         ↓
              用户输入 → agent loop
                         ↓
              LLM 自己决定要不要读 L2/L3/L4
                         ↓
              LLM 觉得对话中有重要信息？
                         ↓
              自觉调用 start_long_term_update 工具
                         ↓
              写入 global_mem.txt / L3 / L4
```

### 1.2 记忆分级


| 层级 | 存储位置                  | 内容                                          | 更新方式                           |
| ------ | --------------------------- | ----------------------------------------------- | ------------------------------------ |
| L1   | `global_mem_insight.txt`  | 存在性指针：告诉 LLM "哪些 L2/L3/L4 资源存在" | 手动编辑                           |
| L2   | `global_mem.txt`          | 用户画像（姓名、偏好、技术栈等）              | LLM 自觉调`start_long_term_update` |
| L3   | `memory/` 下各 SOP 文件   | 技能文档、操作手册                            | LLM 自觉调`start_long_term_update` |
| L4   | `memory/L4_raw_sessions/` | 原始会话存档                                  | LLM 自觉调`start_long_term_update` |

### 1.3 记忆检索

**纯手动**。L1 在系统提示词中提及 L2/L3/L4 的存在，LLM 看到提示词里的钩子后，自己决定要不要去 `file_read` 这些文件。没有任何自动注入机制。

### 1.4 记忆更新

**纯靠 LLM 自觉**。只有一个 `start_long_term_update` 工具，LLM 自己判断什么时候该更新、该更新什么。没有萃取流程，没有强制触发，没有冲突检测。

### 1.5 优缺点

**优点**：

- 系统提示词极短，token 消耗很低，响应快
- 记忆分层清晰（L1 存在性指针 → L2 用户画像 → L3 技能 → L4 历史）

**缺点**：

- L2 用户画像记忆记不牢——LLM 经常忘记调用 `start_long_term_update`，即使对话中明确提到了姓名、偏好等信息
- 即使 LLM 更新了记忆，L1 只有一个钩子名称，LLM 不知道"到底要不要用这个钩子"
- 检索完全依赖 LLM 主动去读文件，跨会话时 LLM 不知道之前记录了什么
- 没有自动化 benchmark，无法量化记忆系统的可靠性

## 2. 改进方案

### 2.1 架构

```
┌─ Session 1 ───────────────────────────────────────────┐
│                                                        │
│  用户输入                                               │
│      ↓                                                 │
│  search_memory(query) ← 关键词模糊匹配 L2               │
│      ↓                                                 │
│  [Auto Retrieved Memory] + 用户输入 → LLM               │
│      ↓                                                 │
│  agent loop (LLM 不再需要自己决定要不要读 L2)             │
│      ↓                                                 │
│  会话结束 (Ctrl+C / sentinel / _stop)                   │
│      ↓                                                 │
│  extract_facts(对话文本 + 现有 L2) ← 强制触发，不靠自觉   │
│      ↓                                                 │
│  auto_update_l2() ← 冲突检测 + 去重：同 key 不同 value 直接替换   │
│      ↓                                                 │
│  global_mem.txt 更新                                    │
│                                                        │
└──────────────────────────────────────────────────────────┘

┌─ Session 2 (新进程) ───────────────────────────────────┐
│                                                        │
│  用户输入: "帮我推荐衣服"                                 │
│      ↓                                                 │
│  search_memory("帮我推荐衣服")                           │
│      ↓                                                 │
│  同义词扩展: "衣服"→"颜色" "风格" "偏好"                   │
│      ↓                                                 │
│  命中 L2: "颜色偏好: 黑色系" "风格偏好: 简约风格"          │
│      ↓                                                 │
│  [Auto Retrieved Memory]                               │
│  - [用户偏好] 颜色偏好: 黑色系                            │
│  - [用户偏好] 风格偏好: 简约风格                           │
│      ↓                                                 │
│  拼接后: "[Auto Retrieved Memory]\n...\n帮我推荐几件衣服" │
│      ↓                                                 │
│  LLM 已自动知道用户偏好，不需追问                          │
│                                                        │
└──────────────────────────────────────────────────────────┘
```

### 2.2 保留的机制

记忆分层结构完全保留：

```
L1 (global_mem_insight.txt) + CONSTITUTION
  ├── L2 (global_mem.txt)          ← 改进重点：自动萃取 + 自动检索
  ├── L3 (SOP 文件)                ← 暂未改动
  └── L4 (L4_raw_sessions/)        ← 暂未改动
```

### 2.3 改动一：会话级强制萃取

**原方案**：LLM 自觉调 `start_long_term_update`，经常忘记。

**改进**：会话结束时强制触发 `extract_facts()`，且只取对话中 `role='user'` 的消息（用户发言才含画像信息，过滤掉 agent 的工具输出噪音）。

```
触发方式：
  - 交互模式 Ctrl+C → sentinel → 萃取 → 退出
  - Task 模式结束 → sentinel → 萃取 → 退出
  - Benchmark _stop 文件 → 检测到 → 退出循环 → 萃取
  - 连按两次 Ctrl+C → 强制退出，放弃萃取
```

**目前仅做了人物肖像（L2）的萃取**，L3/L4 暂未接入。

### 2.4 改动二：自动检索注入

**原方案**：L1 只有钩子名称，LLM 要自己决定去不去读 L2。

**改进**：每次用户输入时，`search_memory()` 自动匹配 L2 中的相关记忆，注入到 query 前面。

**检索算法**：

1. 解析 L2 所有 section + fact 行
2. 从用户 query 提取**中文 2-gram** + 英文单词作为关键词
3. **同义词扩展**：

```
'衣服' → ['颜色', '风格', '偏好', '搭配', '穿着']
'推荐' → ['偏好', '喜欢', '颜色', '风格']
'背景' → ['技术', '职位', '公司', '所在地', '组织']
'前景' → ['技术', '职位', '方向', '领域']
...
```

4. 对每个 fact 打分，返回 Top 5
5. 拼成 `[Auto Retrieved Memory]` 前缀注入

**效果**：LLM 不再需要自己猜测"要不要读 L2"，相关记忆自动出现。

### 2.5 改动三：LLM 感知已有记忆 + 冲突检测

**原方案**：LLM 写记忆时不知道 L2 已有内容，可能产生重复 key（如两条 `颜色偏好`）。

**改进**：两层保障。

**第一层 — LLM 层面**：`extract_facts()` 将现有 L2 内容注入 prompt：

```
=== 当前已存储的事实（参考，避免重复 key） ===
## [用户偏好]
- 颜色偏好: 黑色系
=== 结束 ===

规则：
- 如果新信息与已存储 key 相同 value 不同 → 输出新 value（视为更新）
- 如果完全一致 → 不要重复输出
- 如果用户明确否定了旧偏好 → 只输出新 value
```

**第二层 — 代码层面**：`auto_update_l2()` 做 regex 兜底，相同 key 不同 value 时直接替换旧值，不留历史。

### 2.6 Benchmark


| 用例         | Session 1                          | Session 2          | 验证点               |
| -------------- | ------------------------------------ | -------------------- | ---------------------- |
| CASE-01 偏好 | 自然聊衣服，提"黑色系简约"         | "帮我推荐衣服"     | 自动注入偏好，不追问 |
| CASE-02 身份 | 写代码时提"腾讯后端 Python Docker" | "学什么技能有前景" | 回复结合具体背景     |
| CASE-03 弃权 | 只聊 JSON，无个人信息              | "我的手机号"       | 诚实说不知道         |
| CASE-04 更新 | 已有"黑色系"，说"喜欢白色了"       | "帮我推荐衣服"     | 旧值排除，新值生效   |


| 模式 | 机制                 | 耗时   |
| ------ | ---------------------- | -------- |
| 离线 | 直接调 Python 函数   | <1s    |
| E2E  | 启动 GA 进程真实对话 | ~10min |

**设计原则**：所有对话**无"记住"等显式指令**，信息自然流露。需验证的指标包括萃取存储率、检索命中率、弃权准确率和记忆更新正确性。

## 3. 对比总结


| 维度                     | 原方案                             | 改进方案                                 |
| -------------------------- | ------------------------------------ | ------------------------------------------ |
| **更新触发**             | LLM 自觉调`start_long_term_update` | 会话结束时强制`extract_facts()`          |
| **检索方式**             | LLM 看 L1 钩子后自行决定读 L2      | 每次用户输入自动`search_memory()` 注入   |
| **LLM 是否感知已有记忆** | 否                                 | 是，L2 注入萃取 prompt                   |
| **冲突处理**             | 无                                 | LLM 识别 + regex 兜底，同 key 直接替换   |
| **系统提示词**           | 极短（只有 L1 钩子）               | 增加（注入检索结果），但换来更精准的回复 |
| **记忆可靠性**           | 低，LLM 经常忘记更新               | 高，强制萃取不依赖自觉                   |
| **跨会话能力**           | 弱，钩子只有名字，LLM 不一定读     | 强，相关记忆自动注入                     |

### 核心思路

**保留**记忆分级体系（L1 钩子 → L2 画像 → L3 技能 → L4 历史），**强制化**两个关键环节：

1. **写入不再靠自觉**：会话结束时强制萃取，LLM 必须总结
2. **读取不再靠自觉**：每次用户输入自动检索注入，LLM 被动接收

从"纯手动 RAG"变成"自动 RAG + 强制存储"，让 AI 越来越懂用户，同时保持记忆结构的清晰分层。
